그럼 DTO는 왜 따로 쓰는가?
user 테이블을 직접 조회하더라도,
결과를 받아서 Java 코드로 처리할 땐
AdminDashboardDto 또는 AdminUserStatsDto 등
별도 DTO로 결과를 옮겨 담아서 관리자 페이지에 넘기는 거야.

외래키 사용
한 테이블이 다른 테이블의 기본키를 참조

DTO 구조 (프론트에 보이기 위한 데이터 가공 용)
관리자 페이지는 이런 실제 테이블에서 가져온 정보를 가공해서 보여주는 화면이기 때문에,
조회 목적에 맞게 따로 DTO를 만들어야 해.

예시:
🔸 관리자 대시보드
java
복사
편집
AdminDashboardDto {
  int totalUsers;
  long totalFundingAmount;
  int activeProjects;
  ...
}
🔸 펀딩 통계
java
복사
편집
AdminFundingStatsDto {
  long totalFunding;
  long thisMonthFunding;
  double successRate;
  List<MonthlyFundingTrend> trends;
  ...
}
🔸 회원 상태 통계
java
복사
편집
AdminUserStatusStatsDto {
  int active;
  int pending;
  int suspended;
  ...
}
➡️ 이 DTO들은 **user, project, funding 테이블을 직접 join하거나 조건 조회한 결과를 담는 '뷰 용 그릇'**이야.
➡️ 화면에서 필요한 데이터만 가볍게 담아서 쓸 수 있어.
